System Architecture & Design Specification: DXF Generator
Project: Agentic CAD / AI-Driven House Design Pipeline
Module Owners: Harshit & Devam
Document Status: Final Architecture Spec

1. Executive Summary & Scope
The DXF Generator serves as the final rendering and drafting engine for the generative architectural pipeline. It acts as the bridge between raw mathematical constraints (generated by the Z3 solver and Constraint Agents) and industry-standard human-readable deliverables. It ingests abstract polygon JSON data (both 2D and 3D) and produces perfectly layered, structurally accurate AutoCAD DXF files alongside rich, color-coded PNG visual previews.

1.1. In-Scope Responsibilities
- Geometric Translation: Converts abstract JSON coordinates into strict CAD geometries (Polylines, 3D Polylines, Lines, Text, Dimensions).
- Layer Orchestration: Automatically maps room types and functional areas to standard architectural CAD layers and AutoCAD Color Indexes (ACI).
- Automated Dimensioning: Computes and injects mathematically perfect dimension lines, arrows, and extension lines using custom architectural sizing heuristics.
- Visualization: Renders 2D top-down orthogonal projections (PNG) featuring transparent color washes for human-in-the-loop review.

1.2. Out-of-Scope (Anti-Patterns)
- Geometry Validation: The DXF Generator blindly draws what it is handed. It does not check if rooms overlap, if the house fits on the site, or if coordinates are negative. That is the exclusive responsibility of the Verifier Agent.
- Geometric Repair: The engine does not fix broken polygons or close unclosed geometry loops. It will faithfully render broken coordinates exactly as they are supplied by the Generator Agent.

2. System Architecture: The Modular Subsystem Model
To ensure maximum scalability and future-proofing, the DXF Generator employs a highly decoupled multi-module architecture. This isolates business logic (JSON parsing) from drawing logic (CAD APIs) and rendering logic (Matplotlib).

Layer 1: Universal Ingestion (parser.py)
Role: The strict boundary API between the upstream AI agents and the CAD engine.
Behavior: Parses disparate JSON formats (`floor_plan.json`, `site_plan.json`) and normalizes them into a single, strict internal Intermediate Representation (IR).
Design Note: By enforcing the IR contract, upstream AI agents can change their JSON schema 100 times, and the core CAD engine will never break. Only `parser.py` must be updated to bridge the new JSON into the IR.

Layer 2: Core CAD Engine (core_engine.py)
Role: The low-level execution wrapper for the `ezdxf` library.
Behavior: Manages the DXF ModelSpace, document initialization, layer registration, and abstracts away messy DXF API calls into clean Python functions (`draw_polygon`, `draw_text`).
Design Note: It includes an intelligent 3D detection matrix. If it is handed a 3-tuple `[X, Y, Z]`, it automatically abandons flat AutoCAD `LWPolyline` generation and spawns true spatial `Polyline3D` objects.

Layer 3: Orchestration & Merging (dxf_generator.py)
Role: The main execution loop and file router.
Behavior: Iterates through the IR, spawns instances of the CAD engine, and physically writes the DXF entities.
Design Note: It implements a Layer Prefixing protocol. If multiple JSON files are ingested simultaneously, it prefixes every single generated layer with its `view_id` (e.g., `floor_plan_Room_Kitchen`). This allows infinite JSONs to be merged into a single DXF without layer collisions.

Layer 4: Dimensioning Engine (dimensioning.py)
Role: Mathematical calculation of dimension lines.
Behavior: Computes normals, offsets, and orthogonal vectors to draw parallel dimension lines across all polygons.

Layer 5: Visualization Engine (renderer.py)
Role: The human-review PNG generator.
Behavior: Projects the DXF geometries onto a 2D matplotlib canvas, stripping out the Z coordinate, and applies 10% alpha color washes for aesthetic review.

3. Key Design Choices & Implementation Details

3.2. Two-Pass Z-Index Orchestration
When writing entities to a flat DXF file, draw order dictates visual hierarchy.
Implementation: `dxf_generator.py` uses a two-pass iteration loop.
Pass 1: Iterates through the IR and draws all physical walls, boundaries, and dimension lines.
Pass 2: Iterates through the IR again to draw all text, room labels, and area calculations.
Why: This guarantees that text is stamped on top of all geometric clutter, ensuring it is never accidentally occluded by overlapping walls.

3.3. Architectural Dimensioning Heuristics
AutoCAD's default `STANDARD` dimension style uses raw decimal measurements (e.g., 12.5) and massive 2.5-unit arrows.
Implementation: `dimensioning.py` overrides the `ezdxf` native parameters aggressively.
- `dimasz`: Set to 0.3 (Arrow size reduced by 85%)
- `dimtxt`: Set to 0.6 (Text size scaled for blueprint readability)
- `dimexe` & `dimexo`: Set to 0.1 (Extension lines hug the walls tightly)
- Value Injection: Decimal values are forcefully intercepted and parsed through `format_feet_inches()` to output standard architectural strings (e.g., `12' 6"`).

3.4. Auto-Routing & Sandbox Containment
When executed via CLI, if the user requests a simple output file (e.g., `floor_plan.dxf`), the engine intercepts the path.
Implementation: It dynamically generates a `generated_files/` sandbox inside its own directory and forcefully routes all `.dxf` and `.png` outputs there to prevent polluting the root workspace or mixing binary files with source code.

4. Core Data Schemas (The IR Contract)

4.1 The Intermediate Representation (IR) Schema
The engine relies entirely on this standardized dictionary structure:
{
  "metadata": {"type": "converted_floor_plan"},
  "views": [
    {
      "view_id": "floor_plan",
      "type": "floor_plan",
      "layers": [
        {"name": "Room_Kitchen", "color": 1},
        {"name": "Boundary", "color": 7}
      ],
      "entities": [
        {
          "type": "polygon",
          "layer": "Room_Kitchen",
          "points": [[0,0], [10,0], [10,10], [0,10]],
          "closed": true,
          "auto_dimension": true
        },
        {
          "type": "label",
          "layer": "Labels",
          "text": "Kitchen\n10' x 10'\nArea: 100.0 sqft",
          "position": [5.0, 5.0],
          "height": 0.8,
          "rotation": 0.0
        }
      ]
    }
  ]
}

4.2 Elevations, Side Views, and 3D Support
The DXF Engine is strictly a geometric plotting engine; it does not hardcode "top-down" logic into its core drawing functions. Because of this, the IR schema natively supports Side Views, Elevations, and True 3D without changing the schema structure:
- 2D Elevations (Side Views): If the upstream agent provides an elevation (e.g., a side view of a house), the `[X, Y]` coordinates in the `points` array simply represent Width and Height (instead of Width and Depth). The engine blindly plots the polygon on the 2D canvas, perfectly rendering the side view.
- True 3D Models: The `points` array accepts variable-length coordinate arrays. If an upstream agent provides 3-tuples `[[X, Y, Z], [X, Y, Z]]`, the `core_engine.py` automatically detects the third dimension, disables 2D auto-dimensioning (which is meaningless in pure 3D space), and spawns true spatial `Polyline3D` objects in the DXF file.
