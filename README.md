# Constraint Agent & Entity Constraint Engine - Complete Architecture Guide

## Overview
This document serves as the comprehensive manual for the **Constraint Agent** and its underlying **Entity Constraint Engine**. 

Together, these components act as the "Master Rules Engine" for the generative pipeline. They form the single source of truth for all architectural, spatial, and zoning constraints, ensuring that both the Floor Plan Generator and the Verifier Agent are evaluating designs against the exact same rulebook.

---

## 1. System Integration Flow (Step-by-Step)

Here is exactly what the system does, step-by-step, when integrating into the broader multi-agent pipeline:

1. **Input from Location Zoning Agent:** 
   The process begins when the external Location Zoning Agent generates a dynamic JSON payload mapping environmental properties (`jurisdiction`, `zone`, `offsets`, `max_coverage`, `tree_preservation`). The Zoning Agent saves this JSON to the central database and passes the unique Hash ID to the Constraint Agent.

2. **Constraint Agent (The Aggregator):** 
   The Constraint Agent receives this **Hash ID**. It queries the database to extract the raw zoning data. It then builds a massive dictionary separated into `exterior` and `interior` blocks.

3. **Applying Zone Overrides:** 
   Before building the interior rooms, the Constraint Agent queries the `ZoneSpecificRules` table using the `jurisdiction` and `zone` provided by the Zoning Agent. If it finds specific municipal codes (e.g., Redmond OBAT requires `max_far` of `3.0`, or Bellevue MDR-1 requires `min_significant_tree_diameter_inches` of `6.0`), it seamlessly injects these rules directly into the active dictionary. Unrecognized novel rules are dynamically appended to the `interior` block without requiring code changes.

4. **Entity Constraint Engine (The Execution Layer):** 
   To build the `interior` block, the Constraint Agent queries the database for every known room type (e.g., `bedroom`, `bathroom`, `kitchen`). It loops through them and calls the Entity Constraint Engine. The Engine dives into the `EntitySpecs` and `RelationalRules` tables to extract granular physical limits (aspect ratios), safety limits (egress, ventilation), and complex relational logic (distances, required connectivities, avoidances) for each individual room.

5. **Assembly, Storage & Hashing:** 
   The Constraint Agent aggregates all this data (Exterior zoning limits + Zone specific overrides + Interior entity rules) into a rigid JSON structure conforming to `CONSTRAINT_SCHEMA.md`. It generates a unique SHA-256 hash for this JSON, saves the JSON block back into the `AgentOutputs` database table, and **returns only the new Ruleset Hash ID** (e.g., `ruleset_52ff1cdb7f9c`).

6. **Output to Downstream Agents:** 
   The Floor Plan Generator and Verifier agents use this returned ID to query the database, ensuring they are both generating and evaluating against the exact same frozen ruleset.

## 2. The Unified Database (`src/database.db`)

We use a single, unified local SQLite database (`database.db`) to keep the ruleset extensible, versionable, and instantly queryable by multiple agents.

### The Tables
*   **`Jurisdictions`**: Stores base exterior rules mapped to cities (e.g., default setbacks).
*   **`EntitySpecs`**: The core repository for per-room specs. Columns include min sizes, aspect ratios, light/ventilation codes (`requires_exterior_window`, `ventilation_type`), safety codes (`requires_egress`), and physical features (`requires_door`, `requires_closet`). Every entity is version-tracked.
*   **`RelationalRules`**: Encodes complex geometry requirements between entities (e.g., `must_touch`, `distance_limit`). Every rule has a plain-language `description` attached to it so Reviewer Agent notes stay consistent with the source.
*   **`AreaRules`**: Broad logic statements (e.g., "Living room must be larger than each bedroom").
*   **`ZoneSpecificRules`**: Holds overrides mapped to a specific `jurisdiction` + `zone` pair.
*   **`AgentOutputs`**: The central storage table for all JSON outputs. This stores both the raw inputs from the Location Zoning Agent AND the final assembled JSON schemas generated by the Constraint Agent, keyed by their Hash IDs.

### Initialization Commands
If you ever need to completely wipe and reset the ruleset database, you run two distinct scripts:
*   `db.py`: Initializes the core tables and seeds the real-world building codes (Seattle, Redmond, Bellevue, Bothell, Kirkland).
*   `demodb.py`: A separate script that injects mock Zoning Agent JSONs into `AgentOutputs` for testing purposes.

```bash
cd src
python3 db.py
python3 demodb.py
```
*(This seeds two demo keys: `zoning_3ded7e729f3d` and `zoning_403aa1801269`)*

## 3. Entity Constraint Engine (`src/entity_constraint_engine.py`)

**What it does:** It acts as the internal database reader for the Constraint Agent. It is completely isolated per entity to ensure rules don't bleed across rooms.

**How it works internally:**
1.  When asked for rules regarding a `"bedroom"`, it connects to `database.db`.
2.  It queries `EntitySpecs` to build `size_rules` (min areas, aspect ratios) and `feature_rules` (egress, windows, doors).
3.  It then queries `RelationalRules` to find any adjacency or distance constraint where the `"bedroom"` is involved (either as Entity A or Entity B).
4.  It packages this into a modular dictionary and returns it to the Constraint Agent.

## 4. Constraint Agent (`src/constraint_agent.py`)

**What it does:** It is the aggregator and the public-facing API for this entire component.

### Execution Command
To generate a final ruleset, pass a zoning hashkey into the Constraint Agent. It will compute everything and save the final ruleset JSON silently to the database.

```bash
cd src
python3 constraint_agent.py zoning_3ded7e729f3d
```

## 5. The Universal Extractor (`src/extract_file.py`)

Because all JSONs (both inputs and outputs) are stored in the same `AgentOutputs` database table, we use a single unified extractor script to view the data. It will automatically query the database using the ID and save the formatted JSON output into the `src/json_files` folder for review.

### Execution Command
Pass any hashkey (either a Zoning Agent hashkey or a Final Ruleset ID):

```bash
cd src
python3 extract_file.py zoning_3ded7e729f3d
```
*(This will generate `src/json_files/zoning_3ded7e729f3d.json`)*

## 6. Testing the Entity Constraint Engine

To debug or query the raw baseline rules for a specific entity type (e.g. what are the default rules for a bedroom?), run the engine directly. You can pass multiple entities separated by spaces, or pass `all` to output everything. The outputs are automatically saved to the `src/Entity_Constraints` folder.

*(Currently available entities: `bedroom`, `bathroom`, `kitchen`, `living`, `dining`, `corridor`, `laundry`, `garage`, `balcony`)*

**For a specific room:**
```bash
cd src
python3 entity_constraint_engine.py bedroom
```
*(Generates `src/Entity_Constraints/bedroom.json`)*

**For multiple specific rooms:**
```bash
cd src
python3 entity_constraint_engine.py bedroom bathroom kitchen
```
*(Generates `src/Entity_Constraints/bedroom_bathroom_kitchen.json`)*

**For all rooms at once:**
```bash
cd src
python3 entity_constraint_engine.py all
```
*(Generates `src/Entity_Constraints/all_entities.json`)*

## 7. Extensibility

Because the entire system is DB-driven, new constraints can be added *without touching agent code*:
*   **Adding a new room type:** Simply insert a new row into `EntitySpecs`. The agent will automatically query it and add it to the final JSON.
*   **Adding a new safety code:** Insert a new column into `EntitySpecs`. The verifier can choose to enforce it when ready.
*   **Adding a new zone rule:** Insert a row into `ZoneSpecificRules` for the targeted zone and rule key (e.g. `max_far: 3.0`). The Constraint Agent will dynamically ingest it and append it to the JSON when that zone is detected.

## 8. Natural Language User Constraints

The architecture features a powerful LLM Parser (`src/llm_parser.py`) integrated via the Google Gemini API. It can interpret unstructured natural language requests (e.g. *"I want 3 bedrooms of 100sq ft"*) and automatically map them to strict JSON constraints.

**Key capabilities:**
- Extracts exactly which rooms the user asked for and dynamically adjusts the `required_rooms` baseline.
- Converts conversational area and length constraints into strict mathematical ranges (`±5 sqft`, `±0.5 ft`).
- Handles novel global constraints (e.g. *"max height is 120ft"*) and cleanly merges them without breaking schema.
- Defaults all user-provided rules to `"hard"` constraints unless they specify `"soft"`.

To enable this feature, simply place a `user_constraints.txt` file in the `src` directory, export your Gemini API key (`export GEMINI_API_KEY="..."`), and run `constraint_agent.py`.
